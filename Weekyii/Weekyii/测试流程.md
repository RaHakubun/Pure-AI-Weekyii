# Weekyii 每次功能开发后的测试流程（LLM 可执行版）

## 0. 目标与原则
- 目标：在每次功能开发后，快速发现回归、定位问题、沉淀可复用测试资产。
- 原则：先快后慢、先稳定后脆弱、先自动化后手工链路。
- 统一设备：`iPhone 17 Pro` 模拟器。
- 统一要求：测试结论必须附带可复现命令与失败证据（日志/截图/关键报错）。

---

## 1. 标准执行顺序（必须按顺序）

### Step 1: 基线检查（30秒内）
1. 查看改动范围：
```bash
git status --short
```
2. 确认目标与 scheme：
```bash
xcodebuild -list -project Weekyii.xcodeproj
```

通过标准：
- 能明确本次改了哪些文件。
- 知道要测试的 target/scheme（通常是 `Weekyii`）。

### Step 2: 编译检查（最快失败）
```bash
xcodebuild build \
  -project Weekyii.xcodeproj \
  -scheme Weekyii \
  -destination 'platform=iOS Simulator,name=iPhone 17 Pro'
```

通过标准：
- `BUILD SUCCEEDED`。

失败处理：
- 优先修复编译错误，再进入下一步。

### Step 3: 单元测试（主逻辑）
```bash
xcodebuild test \
  -project Weekyii.xcodeproj \
  -scheme Weekyii \
  -destination 'platform=iOS Simulator,name=iPhone 17 Pro' \
  -only-testing:WeekyiiTests
```

通过标准：
- 单测 100% 通过。

失败处理：
- 按失败用例名定位对应模块（例如状态机、周计算、模型关系），修复后重跑本步骤。

### Step 4: UI 测试（交互回归）
```bash
xcodebuild test \
  -project Weekyii.xcodeproj \
  -scheme Weekyii \
  -destination 'platform=iOS Simulator,name=iPhone 17 Pro' \
  -only-testing:WeekyiiUITests
```

通过标准：
- UI 用例通过。

失败处理：
- 先判断是“业务逻辑问题”还是“测试脆弱问题”（文案/可访问性 id 变更）。
- UI 测试失败不能自动等价为功能坏掉，必须补手工验证结论。

### Step 5: 手工主链路回归（必须）
每次至少覆盖以下路径：
1. 创建当天任务 -> 编辑 -> `start` -> `done focus_zone` -> 完成。
2. 修改 kill time -> 到点过期逻辑。
3. 创建未来日期/未来周 -> 进入当前周时状态迁移。
4. 任务详情可打开（包含已启动、已完成场景）。
5. 手势冲突检查（滑块、横向切换、列表滚动）。

通过标准：
- 主链路无阻塞、无崩溃、状态转换符合设计。

## 2. 测试用例编写规范（单测/UI 通用）

## 2.1 单个用例的标准结构
每个用例必须包含：
1. 前置数据（Given）。
2. 执行动作（When）。
3. 可断言结果（Then）。

推荐命名：
- `test_<模块>_<场景>_<期望结果>`
- 例如：`test_killTime_executeToExpired`

## 2.2 单测编写要求
- 一个用例只验证一个核心行为。
- 避免依赖真实时间，优先注入固定时间。
- 避免依赖真实持久化文件，优先内存容器。
- 断言必须包含：状态值、关键字段、计数统计。

## 2.3 UI 测试编写要求
- 每个关键控件都要有稳定 `accessibilityIdentifier`。
- 避免使用脆弱定位（仅靠中文文案、层级索引）。
- 每个操作后都要显式等待目标状态出现。
- 失败信息要能直接定位控件或页面。

---

## 3. 边界检测清单（每次功能都要对照）

## 3.1 时间与日期边界
- kill time 前后一分钟行为是否正确。
- 跨天（23:59 -> 00:00）状态是否迁移正确。
- 跨周（周日 -> 周一）Present/Past/Pending 是否正确迁移。
- ISO 周校验：仅接受 `W01...W53` 且必须是该年真实周。

## 3.2 数据与关系边界
- 空输入（空标题、空任务列表）是否阻止提交。
- 重复创建（同周、同天）是否幂等处理。
- 删除策略是否符合设计（仅删壳/级联删除）。
- 关系字段（如 task-project/day）是否出现悬挂引用。

## 3.3 交互边界
- 横向手势与局部控件拖动是否冲突。
- 启动后与未启动时，点击行为是否符合规则。
- 长列表/多任务时是否卡顿或错乱。

## 3.4 本地化边界
- 不允许硬编码中文兜底文案。
- 日期格式必须 locale-aware，不写死 `M月d日` 这类格式。
- 中英文切换后 UI 不应错位或截断核心信息。

---

## 4. 功能级测试最小模板（新增功能必补）
每新增一个功能，至少新增以下内容：
1. `>= 1` 个成功路径单测。
2. `>= 1` 个失败/边界单测。
3. `>= 1` 个关键 UI 操作用例（或手工回归记录）。
4. 在 PR/提交说明中写明：
   - 跑了哪些命令。
   - 哪些通过，哪些跳过。
   - 剩余风险是什么。

---

## 5. 通过门禁（Release Gate）
必须全部满足才可判定“可合入/可发布”：
1. 编译通过。
2. `WeekyiiTests` 全通过。
3. 本次改动相关 UI 测试通过，或有明确豁免说明与手工验证证据。
4. 主链路手工回归通过。
5. 无 P0/P1 级已知缺陷。

---

## 6. LLM 执行指令模板（可直接复用）
当任何 LLM 执行测试时，应按下面格式输出：

1. 执行命令清单（逐条）。
2. 每条命令结果：通过/失败。
3. 失败摘要：
   - 失败用例名
   - 关键报错
   - 初步定位文件与行号
4. 本次结论：
   - 是否可合入
   - 剩余风险
   - 下一步建议

建议输出示例：
```text
[TEST] build: pass
[TEST] WeekyiiTests: pass (14/14)
[TEST] WeekyiiUITests: fail (1 case)
[FAIL] DraftReorderUITests.testDragHandleReordersDraftTasks
[CAUSE] draftEditButton identifier not found
[IMPACT] UI regression risk in draft editor entry
[DECISION] not ready to merge
```

---

## 7. Weekyii 专项补充
- 状态机相关改动必须重跑：`StateMachineTests`。
- 周编号/日期相关改动必须重跑：`WeekCalculatorTests`。
- 今日页交互改动必须手工检查：
  - 任务卡片点击详情
  - 截止时间滑块
  - 今日/本周切换手势
- 如果仓库存在重复文件（例如 `Features/*.swift` 与 `Features/Extensions/*.swift`），必须确认实际 target membership，避免“修改未生效”。
